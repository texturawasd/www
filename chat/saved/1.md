When choosing a C standard for a library that must work in **"any" C project** (i.e., maximum backward compatibility across *all* C projects), **C99 is the best choice**. Here's why, with clear reasoning and practical context:

---

### 🔍 Why C99 > C11 for "Any" Project Compatibility
1. **C99 is the *lowest common denominator* standard**:
   - C99 ( ratified in 1999) is **universally supported** by all major C compilers (GCC, Clang, MSVC, Intel C++), including older versions (e.g., GCC 3.x, MSVC 6.0+).
   - C11 (ratified in 2011) adds **significant modern features** (threads, atomics, `_Alignas`, `stdatomic.h`, `threads.h`, etc.) that **many projects avoid** due to:
     - Embedded systems (e.g., microcontrollers) often use older compilers (C89/C90).
     - Legacy projects (e.g., Windows 95/2000 apps) may only support C89.
     - Projects with strict real-time constraints (e.g., aerospace, automotive) often avoid C11 features.
   - **Critical point**: If your library uses *any* C11 feature (e.g., `stdatomic.h`), it **will fail to compile** in projects using C90/C89 or older C11-compatible compilers (e.g., some embedded toolchains).

2. **C99 enables broad compatibility without sacrificing modern usability**:
   - C99 adds **practical features** that most projects need (e.g., `stdarg.h` for variadic macros, `_Complex` types, `restrict` qualifier, `inline` keywords), but **no C11 features**.
   - Projects using C99 (the *most common* modern standard) will **compile your library without modification**.
   - Projects using **older standards** (C89/C90) will still work with your library *if you avoid C11* (see below).

3. **C11 is *not* "any" project compatible**:
   - C11 requires **explicit compiler flags** (e.g., `-std=c11`) in many projects.
   - **Embedded systems** (e.g., ESP32, Arduino) often use C89 or C90.
   - **Legacy Windows projects** (pre-2010) rarely support C11.
   - **Real-world example**: If your library uses `stdatomic.h`, it **will not compile** on:
     - Arduino (uses C89)
     - Older MSVC (pre-2010)
     - Many Linux kernel modules (use C89/C90)

---

### 🛠️ How to Ensure "Any" Project Compatibility (Practical Steps)
To maximize compatibility, **target C99** and avoid C11 features. Here's what to do:

| **Action**                          | **Why It Works**                                                                 |
|--------------------------------------|--------------------------------------------------------------------------------|
| **Use C99 headers** (e.g., `#include <stdio.h>`) | Avoids C11 headers like `stdatomic.h` or `<threads.h>`.                         |
| **Avoid C11 keywords** (e.g., `_Atomic`, `threads`) | Stick to C99 alternatives (e.g., use mutexes, not threads).                     |
| **Use `#define` for older standards** | Example: `#ifdef __STDC_VERSION__` to check for C99 (but **don't use C11**). |
| **Test with C89 compilers**          | Critical! Ensure your library compiles with `gcc -std=c89` (e.g., on older systems). |

> ✅ **Best practice**: Write your library **as a C99-compliant library** (no C11 features). This covers **>99% of real-world projects** (including embedded, legacy, and modern desktop/mobile apps).

---

### 🌐 Real-World Evidence
- **Linux kernel**: Uses C99 (not C11) for most modules.
- **Arduino**: Uses C89 (not C99/C11).
- **Windows legacy apps**: Often use C89/C90.
- **Embedded systems** (e.g., Raspberry Pi, STM32): Most use C89/C90.
- **Industry standards** (e.g., MISRA C): Require C89/C90 for safety-critical systems.

> 💡 **Key takeaway**: If your goal is **"any C project"**, you **must** avoid C11. C99 is the *only* standard that satisfies this without forcing projects to upgrade their toolchains.

---

### 🚫 Why C11 is *Not* the Answer (Even if you think it is)
- **False assumption**: "C11 is newer, so it's better for compatibility" → **No**. Newer standards *reduce* compatibility.
- **C11's "benefits" are irrelevant** for "any" projects:
  - Threads/atomics: Not needed in 90% of projects (e.g., most apps don't need concurrency).
  - `_Alignas`: Only useful in niche performance-critical code.
  - **Result**: C11 **breaks compatibility** with projects that don't support it.

---

### ✅ Final Recommendation
| **Standard** | **Best for "Any" Project?** | **Why**                                                                 |
|---------------|----------------------------|-------------------------------------------------------------------------|
| **C99**        | ✅ **YES**                  | Universally supported, no C11 features, works with C89/C90 projects.      |
| **C11**        | ❌ NO                       | Breaks compatibility with legacy/embedded projects; requires extra flags. |

**Choose C99**. Write your library to **only use C99 features** (e.g., `stdarg.h`, `restrict`, `inline`), and **avoid all C11 features**. This ensures your library works in **every C project** without modification.

> 💡 **Pro tip**: Add a `#define` like `#define C99` to your header to signal users it's C99-compliant (e.g., `#ifdef C99` checks). This gives users confidence it's "any" project compatible.
